chunk至关重要  
`参考：https://zhuanlan.zhihu.com/p/673906072`  RAG 分块Chunk技术优劣、技巧、方法汇总
`参考：https://www.cnblogs.com/informatics/p/19647478`
    总结上述文章：没有最佳的chunk策略，只有合适的；先了解chunk的缺点，影响chunk的因素；选择较为合适的chunk，甚至需要多种chunk策略。

    chunk缺点：上下文信息、位置信息、连续信息、描述信息
                小区域描述具体、关系推理、时间长跨度总结
    影响因素：长中短文档策略不同，Embedding模型选型，用户查询需求目标，描述信息。

### 4、搭建评估系统
    Q：这个场景下，你有没有尝试多生成一些模拟数据或者本身有没有多一些量的业务数据，然后设计一套评估系统来进行检索方案的调试等；

    本身选取了GPT来模拟部分、人工选取部分来组成初步的调试

    比如我要是进行公司内部资料的RAG设计，我在准备好文件的向量化存储后，测试检索召回率，这个应该如何实现？在检索召回率达标之后，
        我需要设计一个测试集：包含部分模拟数据，部分人工制作的数据，合在一起组成最后的评估系统吗？
                模拟数据：eg:扫描文件，例如：“假设你是一名入职三个月的新员工，请针对下面这段行政规定提三个问题。”
                人工：高频问题；易混淆陷阱；否定/超纲问题
            
            检索召回率：设计  “查询-文档”映射表， 比对 ID（计算召回率）计算总的 Hit Rate@5
                {
                "query": "2026年公司的育儿假有多少天？",
                "expected_chunk_ids": ["hr_policy_doc_045", "hr_policy_doc_046"],
                "ground_truth_answer": "根据最新规定，符合条件的员工每年享有10天育儿假。"
                }            

数据更新策略  
    在我实现完整的RAG之后，我有一些新的标准进行了修改，该如何更新在RAG中呢？
            全量重建	删除旧索引，重新构建	数据量小，更新不频繁
            增量更新	只添加新文档/更新变化的文档	数据量大，实时性要求高
            版本管理	每个版本一个索引，切换生效	需要回滚能力
权限控制

评估指标
            指标	含义	如何计算  
            Context Precision	检索到的文档中有多少是相关的	相关文档数 / 检索文档数  
            Context Recall	应该被检索到的文档有多少被找到了	被检索到的相关文档数 / 总相关文档数  
            Faithfulness	回答是否忠实于检索到的文档	LLM-as-Judge 评估  
            Answer Relevancy	回答是否回答了用户的问题	LLM-as-Judge 评估  




### RAG全流程部署：
    设计一个前端来输入用户提问的问题，然后将问题通过fastapi发送到后端的Embedding模型，将Query转为向量，进行向量检索；得到最后组合好的Prompt，，然后投入到Qwen3-0.6B进行推理？
    
        但是这其中vllm起到了什么作用：
            优化部署，提速、增加吞吐量
        Embedding模型 单独部署   ---  解耦结构
            TEI 是由 Hugging Face 开发的一个专门用于部署和提供文本向量化（Embedding）服务的工具。它由 Rust 编写，其设计初衷就是极速。

        连接 Vector DB 执行检索（Top 5），这个也是解耦部署：
            不要把数据库直接塞进 FastAPI 进程里，FastAPI 通过 Client SDK 连接远程向量数据库（如 Milvus, ChromaDB, 或 Qdrant 的服务端模式）

        ChromaDB数据库全流程：
            文档切分        向量化      持久化存储（格式：写入数据：ID, 向量, 原始文本, 元数据(如来源、页码)）
